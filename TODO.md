# Задачи развития проекта

## Подтянуть коммуникации с лидов в pipeline Отказов

**Статус:** ожидает

### Проблема

Сейчас pipeline собирает историю коммуникаций только по сделке через `crm.activity.list` с `ownerTypeId=2, ownerId=<deal_id>`. Всё общение с клиентом до конвертации лида в сделку (звонки, сообщения, комментарии на этапе лида) — теряется. Это искажает анализ, особенно для свежих сделок, где большая часть первичного контакта была на лиде.

### Пример последствий

Сделка 113751 (Ирина) создана 09.04.2026 на основе лида от 08.04.2026. Первый и ключевой звонок клиентки, из которого видно её запрос, объект, ожидания — был на лиде и в БД не попал. Анализ ведётся по неполной картине.

### Что сделать

1. В Bitrix24 у сделки получить поле `LEAD_ID` — это ID исходного лида (может быть `null` для сделок, созданных напрямую).

2. В воркфлоу «Отказ: Сбор данных сделки» в блоке сбора коммуникаций:
   - Если у сделки есть `LEAD_ID` — сделать дополнительный пагинированный вызов `crm.activity.list` с параметрами `filter[OWNER_TYPE_ID]=1, filter[OWNER_ID]=<LEAD_ID>`.
   - Смерджить результаты с коммуникациями по сделке до сортировки по дате и построения журнала.
   - Помечать события с лида в журнале (например префиксом `[ЛИД]` или отдельным полем в `lost_deal_communications`).

3. В таблицу `lost_deals` добавить колонку `lead_id` (`int`, `nullable`) — сохранять для прослеживаемости.

4. В `lost_deal_communications` добавить колонку `source_entity` (`varchar`: `'deal'` / `'lead'`) чтобы различать происхождение события.

5. Проверить что для чатов/сообщений Wazzup механизм аналогичный работает: сообщения чата на этапе лида тоже привязаны к лиду, `crm.activity.list` с `PROVIDER_ID='IMOPENLINES_SESSION'` по лиду вернёт сессии лида — по ним дёргать `imopenlines.session.history.get`.

6. После внедрения — перепрогнать batch по всем текущим сделкам в `lost_deals`, чтобы обогатить их коммуникациями с лидов.

### Проверка на пилоте

После правки — прогнать сделки 111455 (Александр) и 113751 (Ирина), убедиться что теперь видны события 08.04 и более ранние. Сравнить журнал до/после.

### Миграция БД

```sql
ALTER TABLE lost_deals ADD COLUMN lead_id INTEGER;
ALTER TABLE lost_deal_communications ADD COLUMN source_entity VARCHAR(10) DEFAULT 'deal';
```

### Примечание

Аналогичная правка нужна и в проекте Приостановленных — там такая же дыра. Но это отдельное решение.
