# Отказы Б24 — правила для Claude Code

Язык проекта — русский. Детали бизнес-логики и стека — в BRIEFING.md, не дублировать здесь.

## n8n 2.12.3 — ограничения
- `process.env.X` и `$env.X` в Code-нодах **НЕ работают** (N8N_BLOCK_ENV_ACCESS_IN_NODE=true)
- `$env.X` работает только в expression-полях (URL, headers, body HTTP Request, Set, IF)
- Code-ноды не ходят в сеть — HTTP только через HTTP Request ноды
- SplitInBatches v3: сбор результатов только через `$input.all()`
- Webhook ноды: webhookId должен совпадать с path, иначе молчаливый 404
- `n8n_autofix_workflow` ломает typeVersions — не использовать

## Запуск воркфлоу
- Только через `n8n_test_workflow` MCP, **НЕ** через curl/bash (песочница → 403)
- После запуска — **СТОП**, ждать подтверждения пользователя «закончилось»
- Не проверять статус execution самому — это делает пользователь

## Правило: после запуска воркфлоу — стоп

Когда Claude (в чате или CC) запускает воркфлоу через n8n_test_workflow — он НЕ проверяет статус выполнения сам. Никаких:
- n8n_executions list/get
- n8n_get_workflow для «убедиться что активно»
- повторных запусков «на всякий случай»
- попыток диагностики по сообщениям вроде "No response from n8n server" или таймаутам

"No response" от n8n MCP ≠ ошибка воркфлоу. Это таймаут на HTTP-ответ webhook (30 сек), а воркфлоу при этом работает в фоне.

**Единственный источник истины — Саня.** Он смотрит в n8n UI executions и говорит:
- «работает / закончилось / упало»
- какой статус
- что видит в логах, если ошибка

До его сообщения — молчим, ничего не трогаем, БД не дёргаем.

Исключение: Саня сам попросил «проверь execution» или «посмотри логи» — тогда да.

## Принцип минимальных правок схемы
При клонировании проекта (например, Отказы из Приостановленных) БД-схему клонируем 1:1. Добавляем только новые поля специфичные для нового проекта. Существующие поля НЕ переименовываем, даже если имя не идеально подходит — переименование требует синхронной правки кода воркфлоу и резко повышает риск несовпадений. Мёртвые поля оставляем пустыми.

## Секреты
- `service_role_key` и `db_password` — **никогда в чат**, только в `.env` или credentials n8n
- Утечка секрета → немедленная ротация

## Webhook paths
- Префикс `lost-*` обязателен (lost-trigger, lost-batch, lost-transcription, lost-analysis)
- НЕ `reactivation-*` — это проект Приостановленных
